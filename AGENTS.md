# AGENTS.md - Nano Hunter 代理工作指南

## 目的

本文件面向在 `nano-hunter` 仓库中工作的智能编码代理与后续开发 session。
项目当前处于 Godot 4.6 的 2D 类银河恶魔城原型期，开发重心是先把治理基线、文档留痕和第一版可玩垂直切片的入口打稳，再逐步推进玩法实现。

优先做小而准、可验证、可追溯的改动，不要跳过设计与留痕直接扩张实现范围。

## 项目快照

- 引擎：Godot 4.6.x
- 类型：2D 类银河恶魔城
- 当前阶段：`Vertical Slice / 原型期`
- 主要语言：GDScript
- 当前默认可用插件：`godot_mcp`、`gut`
- `better-terrain` 仍保留在仓库中，但当前阶段不启用
- 设计文档目录：`spec-design/`
- 进度文档目录：`docs/progress/`

## 规则优先级

1. 用户直接要求
2. 本项目 `AGENTS.md`
3. 已批准的设计文档与实现计划
4. 通用默认习惯与辅助技能建议

如用户要求、项目文档和默认习惯冲突，以用户要求和本项目文档为准。

## 交互语言约定

- 面向用户的说明、过程更新、提权理由、子代理任务描述和项目文档默认使用中文
- 提交信息继续使用“中文 + English”
- 若工具、系统固定 UI、第三方命令输出或官方术语无法稳定中文化，可保留原文，并在必要时补充中文说明
- 项目规范应优先保证可理解性与一致性，而不是机械追求所有界面元素都中文化

## 工作流：混合版 Superpowers

### 大功能

满足任一条件，按大功能处理：

- 新增或重做核心玩法系统
- 修改角色控制、战斗循环、状态机、敌人 AI、关卡推进、能力门控
- 新增或重做主流程 UI
- 新增项目级目录规范、资源规范、测试规范、存档规范
- 任何明显影响后续开发方向的工作

大功能必须遵循：

1. `brainstorming` 或等效设计确认
2. 设计文档确认或补写
3. 实现计划
4. 实现
5. 验证
6. 更新进度文档

### 小改动

以下工作可按小改动处理：

- 小范围配置调整
- 插件启停和工程整理
- 小型目录修正
- 单脚本的小 bug 修复
- 不影响整体架构的辅助性修改

小改动允许轻量流程，但仍必须：

1. 先写明目标和影响范围
2. 再实施修改
3. 做最小必要验证
4. 更新进度文档

### 强制升级为大功能的条件

即使表面看起来是小改动，只要出现以下情况，也必须升级为大功能流程：

- 需求边界不清
- 会影响多个子系统
- 需要新建多个核心文件或目录
- 可能改变后续玩法方向
- 设计文档与现实实现明显冲突

## 默认开发节奏

本节是对上面“大功能 / 小改动”规则的执行层细化，用来减少 session 间节奏漂移。
它不能替代任务类型判断，只能在类型已经确定后作为默认执行节奏使用。

### 大功能默认节奏

默认按以下顺序推进：

1. `brainstorming` 或等效设计确认
   - 至少明确目标、成功标准、不做项、关键取舍
   - 若用户已经给出清晰边界，可缩短为“设计确认式 brainstorming”，但不能跳过
2. 设计文档确认或补写
   - 写入 `spec-design/`
3. 简短实现计划
   - 写入 `docs/superpowers/plans/`
4. 判定分支策略
   - 阶段型开发默认 `分支 + worktree`
   - 非阶段型但范围较大的任务默认 `仅分支`
5. 实现
   - 主代理先做任务拆分，并主动判断哪些子任务适合交给 `subagent` 或 `multi-agent`
   - 只要写入范围清楚可分离、下一步不强依赖单一结果，就应积极使用代理协作提速
6. 验证
   - 做工程基线验证
   - 跑最接近的自动化测试
   - 必要时补手动试玩或人工复核
7. 文档收口
   - 更新 `docs/progress/status.md`
   - 更新 `docs/progress/timeline.md`
   - 更新当日日志
8. 提交节奏
   - 默认拆成 2-3 个提交点
   - 推荐拆法：`玩法实现` / `测试与验证` / `工程或文档收口`

### 小改动默认节奏

默认按以下顺序推进：

1. 先写明目标和影响范围
2. 默认采用 `仅分支`
3. 实施单个目标改动
4. 做最小必要验证
5. 更新当日日志；必要时补 `status.md` 或 `timeline.md`
6. 默认 1 个提交；如果混入第二个目标，必须拆开

### 默认节奏的决策规则

- `brainstorming`
  - 所有大功能必须使用
  - 边界不清、会影响后续方向的任务必须使用
  - 明确且低风险的小改动、纯工程整理、单点 bugfix 可以不专门使用
- `subagent`
  - 在任务边界清楚、写入范围可分离、下一步不强依赖单一结果时，默认应主动评估并优先使用
  - 不把“是否用了 agent”当作形式化 KPI，但也不应把“主代理单线推进”当作默认美德
- `multi-agent`
  - 当存在 2 个以上互不阻塞、写入范围可分离、且已经有明确计划的子任务时，默认应主动评估并优先并行
  - 不为了“看起来并行”而硬拆任务；只有当拆分后仍易于整合、不会同时改同一批核心文件时才启用
- `worktree`
  - 小任务优先 `仅分支`
  - 阶段型开发优先 `分支 + worktree`
  - 需要长期保留 `main` 现场、隔离 Godot 编辑器现场或让 review / 子代理独立协作时必须用
- 提交次数
  - 大功能默认不少于 2 次提交，除非整个任务极小且当天已完整收口
  - 小改动默认 1 次提交；一旦混入第二个目标，就拆提交或拆分支
- 可推广范围
  - 阶段 3、4、5 直接沿用大功能默认节奏
  - bugfix、配置修正、文档修订沿用小改动默认节奏
  - 中等任务先按小改动判断；只要触发升级条件，就自动切到大功能节奏

## 文档留痕是强制要求

所有开发活动都必须留痕。未更新文档的开发不算完成，未记录验证结果的改动不算真正交付。

### 文档职责

- `spec-design/`
  - 存放设计文档、玩法设计、系统设计
  - 如实现导致设计变化，必须同步回写

- `docs/progress/status.md`
  - 当前项目状态摘要入口
  - 需要和 `docs/progress/timeline.md` 以及当日日志一起阅读

- `docs/progress/timeline.md`
  - 全项目关键事件时间线

- `docs/progress/YYYY-MM-DD.md`
  - 当日详细开发日志

### 记录要求

每次记录至少写明：

- 做了什么
- 为什么做
- 影响了什么
- 验证结果
- 风险或遗留问题
- 下一步建议

### 文档冲突时的判断顺序

1. 当前代码与可运行结果
2. 最新的进度文档
3. 最新的设计文档
4. 更早的聊天上下文

如发现设计与实现偏离，先更新文档再继续推进。

## Subagent / Multi-Agent 与 Delegation Log

- 主代理默认负责设计确认、任务拆分、结果整合与最终交付；在存在可隔离子任务时，应主动考虑交给 `subagent`
- 当存在 2 个以上互不阻塞、写入范围可分离、且已经有明确计划的任务时，推荐优先启用 `multi-agent` 并行
- 适合 `subagent` 的工作包括：独立调查、spec review、quality review、文档整理、小范围实现
- 适合 `multi-agent` 的工作包括：按文件或目录明确分工的并行实现，例如场景、玩家脚本、HUD、验证文档分别推进
- 以下情况不应委派或不应并行：需求未收敛、核心设计仍在变化、多个任务会改同一批文件、当前下一步强依赖单个结果、单点调试尚未定位
- 若一个任务已经满足“边界清楚、写入可隔离、验证可独立”的条件，应优先考虑代理协作，而不是先默认压回主代理单线处理
- 主代理始终负责：设计确认、任务拆分、结果整合、最终验证、对用户交付；子代理只负责被分配的局部任务
- 当前阶段不将固定 agent 名称或固定 roster 写入项目规范；如果未来目录结构、任务模式和验证流程稳定，再考虑可配置、可复用 agent
- 只要使用了重要的 `subagent` 或 `multi-agent` 协作，就必须在当日日志 `docs/progress/YYYY-MM-DD.md` 中增加 `Delegation Log`
- 每条 `Delegation Log` 至少记录：用途、范围、模式、结果、收口
- 很小的探索性委派可以不单独记录，但只要影响实现、评审、验证或最终结论，就必须记录

## 当前阶段与默认目标

本项目当前为 `Vertical Slice / 原型期`。
这一部分属于可演进的阶段性规范，应在里程碑切换时更新，而不是因为零碎小任务频繁改写。

当前默认目标：

- 主场景
- 玩家基础移动
- 跳跃 / 冲刺 / 基础攻击
- 2 个姿态以内
- 2-3 个元素以内
- 最小 HUD
- 最小可回溯关卡验证

其余中后期设计保留在 `spec-design/`，不默认进入实现。

## 阶段推进与可试玩检查点

- 当前第一版可玩原型已完成前 5 个可试玩检查点：`阶段 1 可启动骨架`、`阶段 2 基础移动手感`、`阶段 3 基础战斗手感`、`阶段 4 最小能力差异`、`阶段 5 教程区垂直切片`
- 阶段 5 之后的当前推荐路线已完成前 3 个验证型阶段：`阶段 6 最小真实战斗循环`、`阶段 7 短链路主流程串联`、`阶段 8 系统稳固与内容生产前准备`
- 阶段 8 之后的当前推荐路线调整为更大颗粒度的内容型阶段：`阶段 9 首个小区域内容生产`、`阶段 10 战斗变化与轻量成长循环`、`阶段 11 可交付试玩 Demo 切片`
- 阶段长度按退出条件定义，不按自然时间长短定义
- 每个阶段都必须同时满足：`可看`、`可玩`、`可调`
- 未达到可试玩标准，不进入下一阶段；如果手感或方向明显不对，允许回退到上一阶段继续调整
- 每完成一个阶段，至少更新 `docs/progress/status.md` 中的当前阶段、阶段目标、当前可玩内容、当前可调项、退出条件、当前资产状态、下一阶段
- 当日日志必须记录本阶段的验证结果、调参重点和用户反馈结论
- `阶段 1` 聚焦 `Main.tscn`、`run/main_scene`、测试房间、相机与基础碰撞
- `阶段 2` 聚焦移动、跳跃、落地与基础状态切换
- `阶段 3` 聚焦最小攻击循环、攻击判定与受击反馈
- `阶段 4` 聚焦至少 1 组能力差异的玩法价值
- `阶段 5` 聚焦教程区短流程的串联、节奏和可理解性
- `阶段 6` 聚焦最小真实战斗循环：基础敌人或攻击性障碍、玩家受击 / 生命 / 死亡或重置闭环，以及教程后的第一段实战压力验证
- `阶段 7` 聚焦短链路主流程串联：至少两段房间或区域的顺序推进、最小门控、房间切换或流程衔接
- `阶段 8` 聚焦系统稳固与内容生产前准备：HUD 第二轮、参数数据化、敌人与关卡模板化、以及继续扩内容前需要的工程收口
- `阶段 9` 聚焦首个小区域内容生产：`4-6` 个连续房间、第 `2` 类敌人、遭遇组合、checkpoint / 区域重置点与更正式门控
- `阶段 10` 聚焦战斗变化与轻量成长循环：第 `3` 类敌人或精英变体、`1` 个战斗扩展动作、可选支路、挑战房与最小成长反馈
- `阶段 11` 聚焦可交付试玩 Demo 切片：`8-12` 个房间的完整 demo 区域、阶段性终点、文案 / HUD / 节奏 polish 与 demo 级 QA

## 目录约定

当前项目应逐步形成以下目录：

- `scenes/`：游戏场景
- `scripts/`：核心脚本
- `assets/`：美术与音频资源
- `tests/`：GUT 测试
- `spec-design/`：设计文档
- `docs/progress/`：进度与时间线文档

原则：

- 目录扩展小步前进
- 没有明确需求时，不提前铺设复杂结构
- 命名优先服务当前原型开发

## 验证约定

- 工程配置改动后，优先做一次 `godot --headless --path . --import`
- 不要把 `godot --headless --path . --quit` 作为当前阶段的基线验证命令，因为当前项目还没有定义主场景
- 脚本或场景改动后，优先做最小可运行验证
- 若已有 GUT 测试，则优先运行最接近的测试
- 修 bug 时，优先补最小回归测试
- 未记录验证结果，不应声称“完成”

## 插件约定

当前默认启用插件：

- `godot_mcp`
- `gut`

`godot_mcp` 会在编辑器启动时动态注入它需要的 MCP autoload，退出时再清理，不应把这些临时注入行为当作仓库里永久手写的 `project.godot` 配置。

`better-terrain` 当前保留在仓库中，但在当前阶段不启用。之前的基线验证已经证明它在当前 Godot 4.6 环境下会引入解析和兼容性问题，后续如果重新启用，必须先重新验证。

若后续新增或重新启用插件，必须记录：

- 启用日期
- 启用目的
- 影响范围

## 原型期资产策略

- 不让正式资产生产阻塞玩法验证
- 原型期允许混合使用 `占位资产`、`免费替代资产`、`AI 临时资产`、`正式项目资产`
- `阶段 1-3` 以占位和免费替代为主；AI 资产优先用于角色轮廓、能力识别、HUD 方向、教程区氛围等关键视觉探索
- `阶段 4` 起必须保证能力差异在视觉上可读，即使尚未使用正式美术
- `阶段 5` 允许混合资源完成教程区垂直切片，目标是看得懂、玩得下去、能调整方向，而不是最终美术完成
- 只有当玩法模块已经验证成立且短期内不会频繁推翻时，才开始将临时资源替换为更正式资产

## 代码与注释约定

- 项目内新增或修改的代码注释统一使用中文
- 注释只解释不明显的内容，例如状态切换原因、玩法约束、跨脚本流程、临时兼容处理和容易误解的设计意图
- 涉及玩家控制、战斗判定、能力门控、场景切换等关键玩法流程时，应补充能帮助后续快速理解的引导性注释
- 优先让代码本身通过清晰命名和合理拆分表达含义，不用注释重复“给变量赋值”这类低信息量事实
- 如果旧注释就在当前修改范围内，且不影响语义与工具行为，优先顺手整理为中文；当前改动范围外的旧注释不强制清理
- 本约定仅适用于项目内代码，不要求改写第三方插件、外部依赖或具有特殊语义的文件头
- 优先写小而清晰的 GDScript 文件
- 优先显式类型与清晰命名
- 沿用项目现有模式，不做无关的大重构
- 当前阶段优先服务原型可玩性，不提前实现中后期复杂系统

## 提交约定

- 每次提交围绕一个明确进展点
- 提交信息使用“中文 + English”
- 提交前至少确认：
  - 改动已验证
  - 进度文档已更新
  - 未提交无关噪音文件

若只是中途探索但结果未稳定，可暂不提交，但仍需写入当天日志。

## 分支与合并约定

- `main` 只承载当前稳定、已验证、可试玩的主线结果，不直接作为默认开发分支使用
- 任何会修改玩法、场景、脚本、测试、项目规范、核心配置，或预期超过一次小提交的工作，都应从最新 `main` 先开新分支再开发
- 默认优先使用“仅分支”而不是额外创建 worktree；只要当前任务可以在单一工作目录中安全完成，就不额外增加 worktree
- 以下情况适合只开分支：文档修订、小范围 bugfix、单一目标的短任务、不会长时间占用 `main` 现场的实现
- 以下情况应额外创建 worktree：阶段型开发、需要长期保留 `main` 的可试玩现场、需要并行推进多个独立任务、需要隔离 Godot 导入缓存或编辑器现场、需要让 review 或子代理在独立目录中协作
- 只有极小、低风险、纯整理性质的改动才允许直接在 `main` 上处理；若存在任何范围扩张风险，仍应开分支
- 新分支默认使用 `codex/` 前缀，并尽量让分支名直接对应当前阶段或目标，例如 `codex/stage-2-movement-feel`
- 分支开发期间，`main` 继续保持可验证基线；不应把“尚未稳定的实验中间态”直接写到 `main`
- 合并回 `main` 不按开发时长决定，而按“是否形成稳定里程碑”决定；至少需要同时满足：
  - 当前范围实现已完成
  - 关键验证已通过
  - 进度文档已更新
  - 当前结果已经可以作为新的稳定基线
  - 合并后 `main` 仍能直接试玩，或至少能稳定通过当前阶段验证
- 不需要等 5 个阶段全部完成才合并；每当形成一个稳定、已验证、已留痕的阶段性检查点，就应考虑合并回 `main`
- 合并完成后，如无保留需求，应清理对应分支和 worktree，避免长期堆积过期开发环境
- worktree 物理目录清理按以下顺序执行，不要跳步：
  - 先定位所有直接指向该 worktree 路径的 Godot 编辑器 / 运行实例、VS Code、终端、资源管理器窗口及其他将其作为当前工作目录的进程
  - 再关闭这些进程；若用户已明确授权，可直接代为关闭，不必重复确认
  - 然后再删除 worktree 物理目录；Windows 下若路径较长，优先使用长路径方式删除
  - 删除后必须同时复核 `git worktree list` 与磁盘目录状态，确认 Git 侧和物理目录都已清理完成
- 阶段收口、关闭旧 worktree 或切换到新 session 后，还应确认旧会话遗留的 `godot-mcp-pro` bridge 进程已退出；高置信度经验表明，若旧 bridge 持续占用 `6505-6509`，新开的 Godot 编辑器可能会连到旧会话，导致当前会话里的 `godot_mcp` 工具报 `Godot editor is not connected`
- 若出现“Godot 已启动、`godot_mcp` 插件已注入临时 autoload，但当前会话 MCP 工具仍未连接”的组合现象，优先检查 `6505-6509` 的监听进程、启动时间与是否存在旧 `node.exe` bridge 残留，而不要先假设是当前 worktree 配置错误
- 若已确认旧 `6505-6509` 监听全部释放，但 `godot_mcp` 仍未恢复，不要继续把问题只归因为“旧会话占端口”；下一步应按顺序执行：
  - 先重开当前 AI 会话，让新的 `godot-mcp-pro` bridge 重新建立监听
  - 再确认 `6505-6509` 是否出现新启动时间的 `node.exe`
  - 然后重新打开当前 worktree 的 Godot 编辑器
  - 最后立刻复测 `godot_mcp` 工具
- 高置信度经验表明，只有“旧监听已释放 + 新会话 bridge 已监听 + 当前 Godot 编辑器已重开”三者同时满足时，`godot_mcp` 才会稳定恢复直连；缺任何一环都可能继续报 `Godot editor is not connected`

## 分支操作留痕约定

- 任何重要分支操作都必须留痕，包括：从 `main` 开新分支、创建或清理 worktree、合并回 `main`、放弃分支
- 每次留痕都应明确本次采用的是“仅分支”还是“分支 + worktree”，以及为什么这样选，避免后续 session 误判流程是否走偏
- `docs/progress/timeline.md` 记录关键分支事件，至少写明：
  - 分支名
  - 采用模式：`仅分支` 或 `分支 + worktree`
  - 目标范围
  - 开始或合并日期
  - 关键提交或验证结果
- `docs/progress/YYYY-MM-DD.md` 记录当天更细的分支操作，包括：
  - 为什么开分支或合并
  - 为什么本次需要或不需要 worktree
  - 关联的设计文档与实现计划
  - 合并前后的验证结果
  - worktree / 分支是否已经清理
- 若当天涉及 worktree 清理，还应记录删除前是否已关闭对应 Godot / 终端 / 资源管理器窗口，以及若删除失败，最终确认的占用来源
- 若某次开发未新开分支，而是直接在 `main` 上处理，也必须在当日日志里说明理由，避免后续 session 误判为漏走流程

## Session 接续顺序

新 session 进入项目时，建议按以下顺序读取：

1. `AGENTS.md`
2. `docs/progress/status.md`
3. 最近一篇 `docs/progress/YYYY-MM-DD.md`
4. 相关 `spec-design/` 文档

不要只凭聊天末尾几句直接进入编码。

## 测试文件异常排查约定

- 若 GUT 独立 `-gtest` 触发 Godot 原生崩溃，但同类玩法逻辑看起来正常，优先同时怀疑“测试文件文本状态异常”和“测试逻辑错误”，不要只盯玩法代码。
- 新增或大改 GDScript 测试文件后，优先单独跑一次该文件的 `-gtest`，不要只靠其他 suite 间接覆盖。
- 若出现“同一路径最小版能过、原文件独立入口会崩”的现象，优先整文件重写为干净的 UTF-8 文本，再继续排查。
- 若通过整文件重写恢复正常，必须在当日日志和 `status.md` 中留痕，明确写明这是“高置信度推断”，不是 Godot 原生符号级定点结论。
