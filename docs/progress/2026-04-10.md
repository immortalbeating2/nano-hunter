# 2026-04-10

## Summary

- 把“默认开发节奏”正式写回项目主线文档，避免它继续只停留在聊天上下文中。
- 这次修改只涉及治理规则与进度留痕，不包含新的玩法代码、场景改动或插件启停调整。

## What Changed

- 在 `AGENTS.md` 中新增“默认开发节奏”章节，明确：
  - 大功能默认按“设计确认 -> 设计留痕 -> 简短计划 -> 实现 -> 验证 -> 文档收口 -> 2-3 个提交点”推进
  - 小改动默认按“目标与影响范围 -> 仅分支 -> 单目标改动 -> 最小验证 -> 当日日志 -> 1 个提交”推进
  - `brainstorming`、`subagent`、`worktree` 与提交次数的默认决策规则
- 新建设计文档 `spec-design/2026-04-10-development-cadence-standardization.md`，解释为什么要把阶段 1 与阶段 2 之间收敛出的节奏提升为项目规范。
- 新增实现计划 `docs/superpowers/plans/2026-04-10-development-cadence-standardization.md`，把这次治理修订的范围限定在规则收口与文档同步。
- 更新 `docs/progress/status.md` 与 `docs/progress/timeline.md`，让主线入口文档也能反映这次标准化，而不是只靠分支内文档或聊天记录接续。

## Why

- 阶段 2 的执行体验与阶段 1 不完全一致，暴露出“已有规则存在，但执行层默认节奏没有写清”的问题。
- 如果不把这套节奏写回 `AGENTS.md` 和 `docs/progress/`，后续 session 仍可能重复出现：
  - 是否需要显式 `brainstorming` 的判断不一致
  - `subagent` 使用与否被误解成流程 KPI
  - 提交颗粒度忽大忽小
  - `仅分支` 与 `分支 + worktree` 的选择依据在不同 session 中漂移

## Impact

- 后续阶段 3、4、5 默认按“大功能默认节奏”推进。
- 文档修订、工程整理、配置调整与单点 bugfix 默认按“小改动默认节奏”推进。
- 这次标准化不会改变当前 `main` 的阶段 1 可试玩基线，也不会替代现有的“大功能 / 小改动”分类规则。

## Risks / Follow-up

- 这次修改解决的是“规则显式化”，真正是否稳定生效，还取决于后续 session 是否按 `AGENTS.md -> status.md -> 当日日志 -> 相关 spec` 的顺序接续。
- 当前阶段 2 的玩法开发与 GUI 手动试玩收口仍在独立 worktree `codex/stage-2-movement-feel` 中推进，主线这里只同步治理基线，不代替阶段 2 的验证结论。

## Next Suggestions

1. 后续任何阶段型开发都先按新规则判定：是否属于大功能，是否需要 `worktree`。
2. 阶段 3 开始前，用这套节奏复核一次“攻击放阶段 3、冲刺优先放阶段 4、HUD 和房间系统重构放阶段 5”的边界确认是否仍成立。
3. 后续如果再出现流程漂移，应优先回写 `AGENTS.md` 与 `docs/progress/`，不要让规则再次只存在于聊天上下文。
