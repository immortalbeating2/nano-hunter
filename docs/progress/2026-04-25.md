## Update - 2026-04-25 Godot MCP Worktree Entry Tooling

## Summary

- 针对“每次新开 worktree 都容易遇到旧 bridge 占端口、Godot 编辑器连错会话、MCP 重连不稳定”的重复问题，新增了一套仓库内可复用的 `PowerShell` 工具。
- 本轮没有试图从项目侧“根治” `godot-mcp-pro` 的固定端口机制，而是把进场流程工程化成：
  - 先检查
  - 再修复
  - 再打开当前 worktree 的 Godot 编辑器
- 三个入口脚本都支持 `-DryRun`，便于先验证流程，不直接杀进程或拉起编辑器。

## What Changed

- 新增公共函数文件：
  - `scripts/dev/godot-mcp-common.ps1`
- 新增检查脚本：
  - `scripts/dev/check-godot-mcp.ps1`
- 新增修复脚本：
  - `scripts/dev/repair-godot-mcp.ps1`
- 新增打开当前 worktree Godot 编辑器脚本：
  - `scripts/dev/open-worktree-godot.ps1`
- 新增一键串联入口：
  - `scripts/dev/enter-worktree-godot-mcp.ps1`
- 新增安全修复入口：
  - `scripts/dev/safe-repair-godot-mcp.ps1`

## Why

- 前几轮已经反复确认，当前最容易卡住的不是项目配置本身，而是：
  - 旧 `godot-mcp-pro` bridge 进程残留
  - `6505-6509` 被旧会话占用
  - 新 worktree 的 Godot 编辑器启动顺序不对，导致连到旧 bridge 或没连到当前会话
- 这类问题如果继续靠人工排障，几乎每次新 worktree 进场都要重复做同一组检查。
- 因此，本轮把这件事沉淀成项目内显式工具，而不是继续依赖口头经验。

## Verification

- `& '.\\scripts\\dev\\check-godot-mcp.ps1'`
  - 结果：成功输出当前 worktree 路径、bridge 进程、`6505-6509` 监听、Godot 编辑器进程与编辑器到 bridge 的已建立连接
  - 同时也直接暴露出当前环境里确实还存在多批旧 `node.exe` bridge 残留
- `& '.\\scripts\\dev\\repair-godot-mcp.ps1' -DryRun`
  - 结果：成功预演将要关闭的当前 worktree Godot 编辑器与全部 `godot-mcp-pro` bridge 进程
- `& '.\\scripts\\dev\\open-worktree-godot.ps1' -DryRun`
  - 结果：成功预演将使用 `-e --path <current worktree>` 打开的 Godot 编辑器命令，并输出当前 bridge / editor / established connections 状态
- `& '.\\scripts\\dev\\enter-worktree-godot-mcp.ps1' -DryRun`
  - 结果：成功按顺序串起 `check -> repair -> open` 三步，参数转发与输出均正常
- `& '.\\scripts\\dev\\safe-repair-godot-mcp.ps1' -DryRun`
  - 结果：成功验证“默认只关闭当前 worktree Godot 编辑器，不默认杀 bridge；只有显式 `-ForceKillBridge` 才会清 bridge”
- `check-godot-mcp.ps1` 现已新增 `RecommendedAction` 与 `Reason`
  - 结果：当前环境会直接输出 `ReopenSessionThenForceKillBridge`
  - 原因：当前 worktree Godot 编辑器只连到了“看起来像旧桥”的端口批次，而最新 bridge 批次并未被该编辑器消费

## Recommended Usage

- 新 worktree / 新会话进场时，建议固定顺序：
  1. 优先执行 `scripts/dev/enter-worktree-godot-mcp.ps1`
  2. 如果当前会话的 `godot_mcp` 还活着，默认不要杀 bridge；此时让一键入口维持安全模式
  3. 只有在当前会话本来就坏了、或者明确准备重开会话时，才追加 `-ForceKillBridge`
  4. 如需细分排障，再单独运行 `check / safe-repair / open`
- 如果只是想先看脚本会做什么，不想真的杀旧进程或拉起编辑器：
  - `safe-repair-godot-mcp.ps1 -DryRun`
  - `open-worktree-godot.ps1 -DryRun`
  - `enter-worktree-godot-mcp.ps1 -DryRun`

## Risks / Follow-up

- 这套脚本能显著降低新 worktree 进场的排障成本，但它并没有改变 `godot-mcp-pro` 固定端口区间的底层机制，因此不能从根上消灭“旧会话 bridge 残留”。
- 如果后续仍频繁遇到误连旧会话的问题，可以继续考虑：
  - 再加一个“一键串联 repair + open + check”的上层入口脚本
  - 或把这套顺序补进更显眼的开发文档 / 进场清单
