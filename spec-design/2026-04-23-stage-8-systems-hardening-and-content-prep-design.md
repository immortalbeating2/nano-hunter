# 阶段 8：系统稳固与内容生产前准备设计

## 文档定位

本文档用于记录 `nano-hunter` 在 `阶段 8：系统稳固与内容生产前准备` 的第一轮边界、关键决策与验证目标，确保本轮把当前已经成立的三段短链路、HUD 与基础敌人从“可玩原型”收口为“更适合持续扩内容的稳定工程基线”，而不是继续扩新玩法、新房间或新敌人。

## 背景

截至 `2026-04-23`，项目已经完成：

- 阶段 1：可启动骨架
- 阶段 2：基础移动手感
- 阶段 3：基础战斗手感
- 阶段 4：最小能力差异
- 阶段 5：教程区垂直切片
- 阶段 6：最小真实战斗循环
- 阶段 7：短链路主流程串联

当前 `main` 已具备以下稳定基线：

- `Main` 已稳定推进 `TutorialRoom -> CombatTrialRoom -> GoalTrialRoom`
- 玩家已具备移动、跳跃、地面 `dash`、地面普通攻击、生命、受击、无敌与 `defeated`
- HUD 已能显示步骤标题、提示文本、生命值与 `dash` 状态
- `BasicMeleeEnemy` 已能承担第一轮真实战斗压力
- 阶段 1-7 自动化测试已建立并通过

当前最大的工程缺口已经不再是“有没有内容”，而是“现有内容是否足够稳定，能支撑后续继续扩展而不继续堆脚本级临时逻辑”。因此阶段 8 的目标不是继续加房间或加敌人，而是稳固接口、收口配置与建立可复用模板。

## 本轮目标

本轮固定只做以下内容：

- 优先完成参数数据化
- 完成 HUD 第二轮接口收口
- 把 `BasicMeleeEnemy` 整理为更可复用的敌人模板入口
- 保持当前三段短链路可玩性与节奏不回退
- 保持结果满足：
  - 可看：HUD 与房间上下文表达更稳定
  - 可玩：阶段 7 三段链路仍可直接运行，不因系统收口而断链
  - 可调：玩家、敌人、房间关键参数不再散落在多个脚本里

## 明确不在本轮范围内

本轮明确不做：

- 新能力
- 第二类敌人
- 新房间
- 更长主流程
- 地图系统
- 正式 checkpoint
- 存档
- 资源掉落
- 完整数值系统
- 编辑器工具
- 配置热更新

## 关键实现决策

### 1. 参数数据化优先于模板化

- 本轮第一优先级固定为参数数据化
- 只把当前最关键、最常被继续调的参数收口到只读配置资产
- 不做复杂的数据表、编辑器批处理或热切换系统

这样做的原因是：

- 当前最大的维护成本来自“参数散落”
- 先把参数入口收口，后续 HUD 与敌人模板化才有稳定依赖点

### 2. 数据化只做到只读配置资源

- 推荐使用 Godot `Resource`
- 场景实例在固定入口读取并应用
- 当前阶段不引入 JSON、外部表格或复杂运行时装配器

这样做的原因是：

- 能在最小复杂度下把参数从脚本里抽出来
- 不会为了“未来可能的工具化”提前铺过重基础设施

### 3. HUD 第二轮聚焦接口，不聚焦功能扩张

- 当前 `TutorialHUD` 保留为唯一 HUD 入口
- 本轮不扩菜单、背包、结算 UI 或地图
- HUD 第二轮只做：
  - 读取路径稳定化
  - 房间上下文接口固定化
  - 玩家战斗状态接口固定化

这样做的原因是：

- 当前 HUD 已可用，但接口仍偏原型式
- 本轮要解决的是“后续能不能稳扩”，不是“当前还能多显示什么”

### 4. 模板化只先落在敌人侧

- 本轮模板化对象固定为 `BasicMeleeEnemy`
- 不先重做房间模板体系
- 不先重写 `Main`

这样做的原因是：

- 当前最容易在继续扩内容时复制粘贴失控的是敌人逻辑
- 先把敌人模板收口，比先重写房间系统收益更高、更稳

### 5. 现有三段主流程视为稳定前置

- `Main`
- `TutorialRoom`
- `CombatTrialRoom`
- `GoalTrialRoom`

这些都视为当前稳定前置，不在本轮推翻重写。

这样做的原因是：

- 阶段 8 是“系统稳固”，不是“再做一轮主流程重构”
- 若此时重写链路，很容易把范围重新拖回阶段 7

## 预期接口收口方向

### 玩家侧

推荐提供稳定只读接口，例如：

- `get_current_health() -> int`
- `get_max_health() -> int`
- `is_dash_ready() -> bool`
- `get_hud_status_snapshot()` 或等价最小读取接口

### 房间侧

推荐固定 HUD 上下文接口，例如：

- `get_current_step_title() -> String`
- `get_current_prompt_text() -> String`
- `is_dash_available_in_hud() -> bool`

### 敌人侧

继续保留统一基础契约：

- `receive_attack(hit_direction: Vector2, knockback_force: float) -> void`
- `defeated` 信号

### 配置侧

本轮推荐最小拆分为：

- `PlayerConfig`
- `BasicEnemyConfig`
- `RoomFlowConfig` 或按房间拆分的最小配置资源

## 代理协作要求

本轮存在明确的并行空间，计划中不应只写“推荐”，而应在实现时优先实际启用：

- 代理 A：参数数据化与配置资源接入
- 代理 B：HUD 第二轮接口收口
- 代理 C：基础敌人模板化、Stage 8 GUT 与文档留痕

只有在以下条件成立时，才允许不启用 `multi-agent`：

- 写入范围无法隔离
- 核心接口改动成为单点阻塞
- 当前会话工具授权边界仍不允许实际启用

若未启用，必须在 `Delegation Log` 中明确留痕。

## 验证标准

本轮至少需要验证：

1. 玩家关键参数已从配置资源读取并成功应用
2. `BasicMeleeEnemy` 关键参数已从配置资源读取并成功应用
3. 房间/HUD 上下文能通过统一接口稳定读取
4. HUD 不再依赖零散探测式读取来拼核心状态
5. 当前近战敌人模板契约不回归
6. 阶段 1-7 自动化测试继续保持绿色

## 风险与注意项

- 本轮最容易失控的方向是：把“系统稳固”借题发挥成“继续扩玩法内容”
- 若参数数据化和模板化同时扩得过深，容易提前进入复杂基础设施阶段
- 若 HUD 第二轮同时改表现和结构，容易把最小收口做成大改版
- 清理本轮 worktree 时，仍需遵守 `AGENTS.md` 中已写明的顺序：先定位并关闭指向该 worktree 的 Godot / VS Code / 终端等进程，再删物理目录
