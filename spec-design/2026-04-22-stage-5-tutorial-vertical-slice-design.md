# 阶段 5 教程区垂直切片设计

## 文档定位

本文档用于记录 `nano-hunter` 在 `阶段 5：教程区垂直切片` 的第一轮边界、关键决策与验证目标，确保本轮把阶段 2 到阶段 4 已成立的基础能力串成一个最小可玩的主流程，而不是提前扩成完整关卡系统或完整战斗前台系统。

## 背景

当前 `main` 已具备以下稳定基线：

- 阶段 2 的基础移动、跳跃、落地与调参窗口已成立
- 阶段 3 的最小攻击闭环已成立：`attack`、固定命中范围、`TrainingDummy`、最小受击反馈
- 阶段 4 的最小能力差异已成立：仅地面 `dash` 同时具备探索与战斗接敌价值
- 阶段 1 / 2 / 3 / 4 自动化验证保持绿色

同时也已经暴露出一个现实结构约束：

- `Main` 当前仍直接实例化 `TestRoom`
- `Main` 当前仍依赖固定节点名 `TestRoom`
- Stage 1 现有测试也把 `Main -> TestRoom` 当作固定约束

因此阶段 5 不是只“新建一个教程场景”那么简单，而是要把主线入口从“测试房间专用入口”迁移为“主房间契约入口”，同时保留 `TestRoom` 作为回归与调参沙盒。

## 本轮目标

本轮阶段 5 固定只做以下内容：

- 新建专用 `TutorialRoom`
- 将 `Main` 迁移为依赖统一主房间契约，而不是硬编码 `TestRoom`
- 用单场景线性流程串起 `move / jump / dash / attack`
- 新增最小 HUD：
  - 教学提示区
  - 战斗面板区
- 采用低压教学门控，而不是正式死亡 / 恢复循环

同时保持结果满足：

- 可看：玩家第一次进入主流程时能读懂当前目标和按键提示
- 可玩：玩家能顺着教程段稳定完成基础能力学习
- 可调：教程顺序、门控位置、提示内容和 HUD 表达可以继续微调

## 明确不在本轮范围内

本轮明确不做：

- 多房间教程区
- 正式敌人 AI
- 完整生命值消耗、死亡与恢复循环
- 房间系统大重构
- 空中冲刺
- 无敌帧
- 攻击取消
- 连段
- 姿态切换
- 元素系统
- 正式战斗数值系统

## 关键实现决策

### 1. 主流程改用专用 TutorialRoom

- 不把 `TestRoom` 继续演进成主线教程区
- `TestRoom` 保留为阶段 2 / 3 / 4 的回归与调参沙盒
- `Main` 默认加载 `TutorialRoom`

这样做的原因是：

- 避免把“回归沙盒”和“主流程房间”绑死
- 保持早期测试房间的用途清晰
- 为后续主流程扩展留出更干净的入口

### 2. Main 迁移为“主房间契约”而不是“TestRoom 约定”

- 主房间节点应采用通用命名，例如 `Room`
- 主房间继续暴露 `get_camera_limits() -> Rect2i`
- `Main` 继续负责玩家生成与相机限位，但不再写死具体房间类型

这样做的原因是：

- 当前真正需要被主场景依赖的是“房间契约”，不是“测试房间名字”
- 阶段 5 若不先做这层迁移，后续每一轮都要带着 `TestRoom` 的历史耦合前进

### 3. 教程区固定为单场景线性低压教学

- 教程段顺序固定为：
  - 移动 / 跳跃
  - `dash`
  - 攻击
  - 出口
- 每段只承担一个主教学目标
- 门控采用低压模型：
  - 不做正式失败重来
  - 允许玩家原地重复尝试
  - 阻挡目标或障碍只做轻量反馈

这样做的原因是：

- 当前阶段目标是“第一次教会玩家”，不是“第一次惩罚玩家”
- 单场景更适合快速验证节奏与可理解性

### 4. HUD 只做最小前台形态

- 新增最小 HUD，固定包含：
  - 教学提示区
  - 战斗面板区
- 战斗面板只做展示：
  - 可显示满状态生命框位
  - 可显示 `dash` 可用性或能力图标高亮
- 不在本轮接入正式扣血、死亡与恢复流程

这样做的原因是：

- 阶段 5 需要让玩家第一次看到“主流程 UI 长什么样”
- 但当前还没有稳定到值得把完整战斗前台系统一并做完

### 5. 战斗教学继续沿用最小受击契约

- 本轮不引入正式敌人 AI
- 战斗段优先复用 `TrainingDummy` 契约，或做一个同契约的教程阻挡目标
- 继续沿用：
  - `receive_attack(hit_direction: Vector2, knockback_force: float) -> void`

这样做的原因是：

- 阶段 5 要验证的是“流程中的第一次攻击教学”
- 不是“敌人系统是否成熟”

## 协作建议

按当前项目 `AGENTS.md` 的修正版规则，本轮应主动评估代理协作，而不是默认主代理单线推进。

若进入实现阶段后写入范围仍清楚可分离，优先考虑：

- 代理 A：`Main` 与主房间契约迁移
- 代理 B：`TutorialRoom`、HUD 与教程门控
- 代理 C：Stage 1 迁移测试、Stage 5 GUT 与文档收口

若采用重要的 `subagent` 或 `multi-agent` 协作，必须在当日日志中补 `Delegation Log`。

## 验证标准

本轮至少需要验证：

1. `Main` 默认加载教程主房间，而不是继续固定 `TestRoom`。
2. 主房间继续满足 `get_camera_limits() -> Rect2i` 契约。
3. HUD 默认可见，并显示第一条教程提示。
4. 教学步骤按“移动/跳跃 -> dash -> 攻击 -> 出口”顺序推进。
5. 无 `dash` 时不能稳定通过教程门槛。
6. 使用 `dash` 后可以稳定通过教程门槛。
7. 命中教程阻挡目标后，出口或下一段门控解锁。
8. 战斗面板存在，但只以最小展示模式运行。
9. 阶段 1 / 2 / 3 / 4 自动化测试保持绿色。

## 风险与注意项

- `Main` 与 Stage 1 测试当前都还带着 `TestRoom` 历史耦合，阶段 5 需要一次性收口，不应只改一半
- 阶段 5 最容易失控的方向是把 HUD、敌人、房间切换和完整战斗系统提前混进来
- 当前桌面线程内置 `mcp__godot_mcp_pro__` 连接仍可能不稳定；需要运行态复核时优先使用 Godot MCP Pro CLI
- 清理本轮 worktree 前，需先关闭所有指向该 worktree 的 Godot / 终端 / 资源管理器窗口，避免 Windows 占用导致目录删除失败
