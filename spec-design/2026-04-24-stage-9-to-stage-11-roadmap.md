# 阶段 9-11 路线设计

## 文档定位

本文档用于记录 `nano-hunter` 在完成阶段 1-8 之后的当前推荐开发路线，作为后续 `stage9-11` 设计与实现的总入口。它不是直接替代每个阶段的独立设计文档，而是用于明确“为什么下一步这样切、为什么阶段颗粒度要放大”。

## 当前状态

截至 `2026-04-24`，项目已经完成：

- 阶段 1：可启动骨架
- 阶段 2：基础移动手感
- 阶段 3：基础战斗手感
- 阶段 4：最小能力差异
- 阶段 5：教程区垂直切片
- 阶段 6：最小真实战斗循环
- 阶段 7：短链路主流程串联
- 阶段 8：系统稳固与内容生产前准备

当前 `main` 已具备：

- 三段稳定主流程：`TutorialRoom -> CombatTrialRoom -> GoalTrialRoom`
- 玩家移动、跳跃、地面冲刺、基础攻击、生命、受击与重置
- `BasicMeleeEnemy` 最小战斗压力模型
- 最小 HUD
- 玩家、房间与基础敌人的只读配置资源入口
- 基础敌人模板入口
- 阶段 1-8 自动化测试

## 为什么要调整阶段 9-11 的颗粒度

到阶段 8 为止，项目更偏“验证型里程碑”：

- 先验证玩法是否成立
- 再验证主流程是否成立
- 再验证系统接口是否足够稳定

这种切法在原型前半段是合适的，但如果阶段 9 以后还继续按同样大小切分，就容易出现：

- 一个阶段的 worktree 很快完成
- 很快又要开下一个分支 / worktree
- 阶段很多，但每个阶段内部承载的内容偏少
- 内容生产的连续性弱于治理节奏

因此，阶段 9-11 不再沿用“单个系统验证”颗粒度，而改为“一个更完整的可玩成果”作为阶段单位。每个阶段内部允许再拆小里程碑，但原则上放在同一条分支 / worktree 内推进。

## 调整后的推荐路线

### 阶段 9：首个小区域内容生产

目标：

- 用阶段 8 已经建立的配置资源、HUD 接口和敌人模板，真正产出第一段“像游戏区域”的内容，而不是继续只做系统收口。

建议范围：

- 做出 `4-6` 个连续房间的小区域
- 新增第 `2` 类敌人
- 做出 `2-3` 组明确不同的遭遇组合
- 新增第一个正式 checkpoint / 区域重置点
- 新增一种更正式的门控类型
- 继续沿用当前 HUD，并补上与 checkpoint / 门控相关的最小可读性提示
- 把房间配置和敌人配置真正用在这批新内容里，而不是只停留在模板存在

明确不做：

- 大地图
- 正式存档系统
- 多区域世界结构
- 大规模 UI 重做

### 阶段 10：战斗变化与轻量成长循环

目标：

- 在已有小区域之上，让玩法不再只是“过门 + 打一种敌人”，而开始出现战斗变化、路线变化和轻量成长反馈。

建议范围：

- 新增第 `3` 类敌人，或一个精英变体
- 新增 `1` 个玩家战斗扩展动作
  - 默认建议优先为 `空中攻击`
- 新增轻量成长循环：
  - 恢复点 / 恢复物
  - 简单收集物或区域内计数目标
- 小区域中加入 `1` 条可选支路
- 做出 `1` 个可选挑战房或额外奖励房
- HUD 第三轮只补成长相关的最小信息，不做复杂背包或正式属性页

明确不做：

- 正式经济系统
- 装备系统
- 技能树
- 完整存档 / Meta progression

### 阶段 11：可交付试玩 Demo 切片

目标：

- 把前面的系统和内容收成一个可以对外试玩、可连续体验、完成度明显更高的 demo 版本。

建议范围：

- 扩成 `8-12` 个房间的完整 demo 区域
- 至少包含：
  - 引导入口
  - 常规战斗段
  - 平台 / 门控段
  - 支路挑战段
  - 一个阶段性终点
- 完成试玩闭环：
  - 开始
  - 推进
  - 失败 / 重来
  - 完成反馈
- 做一轮更正式的 HUD / 可读性 / 文案 polish
- 做一轮关键视觉替换
- 做一轮 balance pass 与 demo 级 QA / 回归

明确不做：

- 大世界
- 完整剧情系统
- 正式商业化体量的内容量

## 当前默认顺序

当前推荐顺序固定为：

1. `阶段 9：首个小区域内容生产`
2. `阶段 10：战斗变化与轻量成长循环`
3. `阶段 11：可交付试玩 Demo 切片`

## 执行建议

- 阶段 9-11 继续默认采用 `分支 + worktree`
- 但每个阶段内部允许拆 `A/B/C` 三个小里程碑，不再因为阶段太小而频繁开新 worktree
- 如果后续真人试玩或内容生产结果表明阶段划分还需调整，应先更新本路线文档与进度文档，再进入实现
