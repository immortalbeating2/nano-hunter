# Nano Hunter 分支与 Worktree 策略设计

## 文档定位

本文档用于补充 `nano-hunter` 当前阶段的开发工作流，明确“什么时候只开分支，什么时候需要额外创建 worktree”，并作为 [AGENTS.md](D:\Desktop\Project\Game\nano-hunter\AGENTS.md) 中长期执行规则的设计依据。

## 背景

项目在 `阶段 1` 中已经完整走过一次 `分支 + worktree -> 验证 -> 合并回 main -> 清理环境` 的流程，这证明 worktree 对阶段型开发有效；但如果后续所有小改动都默认额外创建 worktree，会让流程变重，也会增加目录和环境管理负担。

因此需要把当前策略收敛为：
- 默认先开分支
- 只有在确实需要现场隔离时再增加 worktree
- 并把每次选择的原因写入进度文档，保证后续可追溯

## 目标

- 保持 `main` 作为稳定、已验证、可试玩的主线
- 减少不必要的 worktree 使用，避免小任务流程过重
- 在阶段开发、并行开发和 Godot 现场隔离需要时，继续保留 worktree 的优势
- 让后续 session 能从文档中看出每次为何只开分支，或为何额外开了 worktree

## 方案对比

### 方案 A：所有非 main 开发都使用 `分支 + worktree`

- 优点：隔离最强，主线目录最稳定
- 缺点：对小任务太重，目录和清理成本高

### 方案 B：永远只开分支，不再使用 worktree

- 优点：流程最轻，目录最简单
- 缺点：阶段开发、并行开发、Godot 编辑器现场隔离时容易互相干扰

### 方案 C：默认只开分支，按需增加 worktree（推荐）

- 优点：小任务轻量，阶段任务仍可隔离
- 缺点：需要在文档中明确判断标准，否则容易重新回到“靠感觉决定”

推荐采用方案 C，因为它最符合 `nano-hunter` 当前阶段：既有持续阶段开发，又不希望每次小修订都引入额外工作目录。

## 执行规则

### 默认规则

- 任何超出极小整理范围的工作，先从最新 `main` 开新分支
- 默认不额外创建 worktree，除非本次任务满足下述条件

### 适合“仅分支”的任务

- 文档修订与规范同步
- 小范围 bugfix
- 单一目标的短任务
- 不需要长时间保留 `main` 试玩现场的改动
- 不需要并行保留多个独立开发目录的工作

### 适合“分支 + worktree”的任务

- 新阶段开发，例如 `阶段 2`、`阶段 3` 这类可试玩检查点推进
- 需要长期保留 `main` 的稳定试玩现场，同时并行推进下一阶段
- 需要多个独立任务并行推进，且写入范围可分离
- 需要隔离 Godot 导入缓存、编辑器状态或运行现场
- 需要让主代理与子代理在独立目录中协作

## 文档留痕要求

### `AGENTS.md`

写入长期规则：
- 默认先开分支
- 何时额外开 worktree
- 合并与清理标准

### `docs/progress/timeline.md`

对每次重要分支操作至少记录：
- 分支名
- 本次采用模式：`仅分支` 或 `分支 + worktree`
- 目标范围
- 开始或合并日期
- 关键提交或验证结果

### `docs/progress/YYYY-MM-DD.md`

对当日分支操作补充说明：
- 为什么开分支
- 为什么本次需要或不需要 worktree
- 关联设计文档与实现计划
- 验证结果
- 是否已清理分支与 worktree

## 当前落地决定

从本次规则更新开始：
- 小范围规则修订优先采用“仅分支”
- 新阶段开发默认采用“分支 + worktree”
- 如果未来阶段开发节奏变化，再回写 `AGENTS.md` 和进度文档

## 结论

`nano-hunter` 当前采用：
- 小任务：开分支
- 阶段任务：开分支并按需创建 worktree

这个标准既保住 `main` 的稳定基线，也避免把所有开发都做成重流程。
