# 阶段 11：可交付试玩 Demo 切片设计

## 目标定位

`stage11` 的目标不是继续扩新系统，而是把 `stage1-10` 已验证成立的内容收成一个可以连续试玩、失败重来、到达终点并得到明确反馈的 demo 级切片。

当前默认做法是“扩成完整 demo”，但扩展主要来自对现有链路的收束，而不是新开第二个大区域或继续叠新机制。

## 当前基线

截至 `2026-04-24`，主线已经具备：

- 教程区三段基础链路
- `stage9` 的五房间线性主线区
- `stage10` 的主房、可选支路与挑战房
- 玩家移动、跳跃、地面攻击、空中攻击、dash
- 三类普通敌人
- checkpoint、门控、HUD 稳定快照
- 最终人工复核通过的 stage10 稳定基线

因此 `stage11` 不再验证“系统是否存在”，而是验证“这些系统是否已经能组成一条完整 demo 体验”。

## 关键设计选择

### 1. Demo 结构

本轮默认采用“复用现有内容 + 新增最小终点表达”的方案：

- 保留：
  - `TutorialRoom`
  - `CombatTrialRoom`
  - `GoalTrialRoom`
  - `stage9_zone_entry_room`
  - `stage9_zone_combat_room`
  - `stage9_zone_charger_room`
  - `stage9_zone_switch_room`
  - `stage9_zone_final_room`
  - `stage10_zone_aerial_room`
  - `stage10_zone_branch_room`
  - `stage10_zone_challenge_room`
- 新增：
  - `1` 个 demo 终点房或等价终点状态

这样当前 demo 将形成 `11 + 1` 的完整试玩链路，仍处于“8-12 个有效体验节点”的目标范围内。

### 2. 本轮不新增机制

本轮固定不做：

- 第 `4` 类敌人
- 新的核心战斗动作
- 第二个大区域
- 正式经济 / 装备 / 技能树
- 完整剧情系统

当前所有工作都应围绕“把现有内容收成 demo”展开，而不是把 stage11 再做成 stage10.5 的系统实验场。

### 3. 可读性与注释

本轮把“代码可读性”提升为显式要求：

- `stage11` 修改过的核心脚本必须满足新的注释最低覆盖标准
- 新增脚本必须有文件头职责注释
- 主流程控制、demo 终点推进、HUD 目标反馈与房间串联逻辑必须有分段注释
- 测试文件必须说明“该测试保护 demo 的什么能力”

这不是额外附带项，而是本阶段完成定义的一部分。

## 预期成果

阶段 11 收口时，应形成：

- 一条可从开始推进到 demo 终点的完整试玩链路
- 至少一次支路进入的价值
- 至少一次挑战房进入的价值
- 失败 / 重来仍可成立
- 终点反馈明确
- HUD / 门控 / 提示足以支撑外部试玩理解

## 风险边界

当前最容易失控的方向有三类：

- 把 demo 闭环做成“继续扩内容”
- 把 HUD polish 做成第三轮系统重构
- 把注释补强拖到最后，导致实现期继续在低可读性基线上前进

因此本轮默认优先级固定为：

1. demo 闭环
2. 可读性与目标提示
3. 注释补强
4. 平衡与人工复核
