# Nano Hunter 默认开发节奏标准化设计

日期：2026-04-10

## 背景

阶段 1 与阶段 2 的推进方式出现了体感差异：

- 阶段 1 的设计、计划、实现、验证、提交流程更清晰
- 阶段 2 虽然最终仍回到了 `AGENTS.md` 要求的留痕框架，但中途更容易出现节奏漂移
- `brainstorming`、`subagent`、提交颗粒度、`worktree` 使用时机在不同 session 之间缺少统一执行口径

项目已经在 `AGENTS.md` 中定义了大功能 / 小改动分流、分支与 worktree 规则、验证约定和留痕要求，但缺少更靠执行层的“默认开发节奏”。

## 目标

把现有规则细化成一套能直接落地的默认节奏，使后续 session 在不改变项目治理原则的前提下：

- 先判断任务类型，再套用默认流程
- 大功能默认先做设计确认、再做计划、再做实现和收口
- 小改动默认保持轻量，但不跳过目标说明、验证和日志
- 统一 `brainstorming`、`subagent`、`worktree` 和提交节奏的使用口径

## 设计结论

### 1. 不新增新体系，只细化现有体系

本次不引入新的任务分类，也不替代现有 `AGENTS.md` 中的：

- 大功能 / 小改动分流
- 分支与 worktree 约定
- 文档留痕要求
- 验证约定

本次只新增“默认开发节奏”，作为这些规则的执行层细化。

### 2. 大功能固定为八步节奏

大功能默认流程固定为：

1. `brainstorming` 或等效设计确认
2. 设计文档确认或补写
3. 简短实现计划
4. 判定分支策略
5. 实现
6. 验证
7. 文档收口
8. 2-3 个提交点收口

其中：

- 当用户边界已经明确时，`brainstorming` 可以缩短，但不能完全跳过
- 阶段型开发默认 `分支 + worktree`
- 主代理单线推进仍是默认值

### 3. 小改动固定为六步轻量节奏

小改动默认流程固定为：

1. 写明目标和影响范围
2. 默认 `仅分支`
3. 实施单个目标改动
4. 做最小必要验证
5. 更新进度日志
6. 默认 1 个提交

如果任务开始混入第二个目标，则不再按轻量节奏收口，必须拆提交、拆分支或升级流程。

### 4. `subagent` 和 `multi-agent` 继续保持条件触发

本次不把 `subagent` 变成默认动作。2026-04-27 起，项目已将可复用游戏项目角色池迁入官方加载路径 `.codex/agents/`，并通过 `.codex/config.toml` 的 `[agents]` 注册；该角色池只是降低后续分工成本，不改变“按任务结构决定是否启用”的原则。

只有在以下条件同时满足时才推荐使用：

- 边界清楚
- 写入范围可分离
- 不会同时改同一批核心文件
- 不阻塞当前关键路径

也就是说，是否使用 agent 仍然由任务结构决定，而不是由流程形式决定。当前默认并行参数为 `max_threads = 4`、`max_depth = 1`，单次任务通常启用 `2-4` 个最相关角色；核心角色 6 个，另有 `godot_runtime` 与 `asset_direction` 作为本项目专项角色按需启用。

### 5. 提交颗粒度统一为“够清晰，不求过碎”

本次不要求所有大功能都像阶段 1 一样高频提交。

统一口径为：

- 大功能默认 2-3 次提交
- 小改动默认 1 次提交
- 不允许长时间把所有内容堆成一个未收口的大改动

## 影响

- 后续阶段 3、4、5 直接沿用大功能默认节奏
- 后续 bugfix、配置修正、文档修订沿用小改动默认节奏
- 中等任务先按小改动判断，只要触发升级条件就自动切换到大功能节奏

## 不做项

- 不要求每个任务都全量启用固定 agent roster
- 不要求每个任务都必须用 `subagent`
- 不要求所有大功能都拆成阶段 1 那种碎提交
- 不修改当前阶段目标划分
