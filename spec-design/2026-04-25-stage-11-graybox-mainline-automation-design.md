# Stage 11 灰盒主线自动化通关设计

日期：2026-04-25

## 背景

当前 `stage11` 已经具备：

- `Main.tscn` 到 `stage11_demo_end_room` 的主线房间链路
- 覆盖终点、支路、完成反馈与失败重来的 Stage 11 GUT
- 可恢复联通的 `godot_mcp`

但当前仍缺一条“非真人、可重复”的完整主线自动化通关验证。现有 MCP 输入手段存在两类问题：

- `simulate_action` 在本项目里会出现明显粘键，不适合长链路 2D 平台动作
- 纯黑盒输入在教程跳台、房间切换与 stage9/stage10 长链路推进中稳定性不足

因此本轮目标不是伪装真人试玩，而是补一条“灰盒主线自动化通关”链路：只靠真实输入推进，但允许读取运行态信号做导航判断。

## 目标

新增一套 Stage 11 灰盒主线自动化验证，满足以下目标：

- 从 `Main.tscn` 的教程起点启动
- 不直接调用切房、瞬移、改血量或改玩家状态接口
- 只通过真实输入推进：`move_right` / `jump` / `attack` / `dash`
- 允许读取运行态房间、HUD 与玩家坐标来决定下一段输入
- 最终真实推进到 `stage11_demo_end_room`
- 触发 demo 完成反馈
- 失败时输出“卡在什么房间、什么步骤、最后玩家状态”

## 不做项

本轮明确不做：

- 不把该自动化扩成“完全黑盒、零内部读数”的真人替代品
- 不修改现有玩法规则来迁就自动化
- 不新增正式录像回放系统
- 不实现通用 AI 导航系统
- 不覆盖支路、挑战房、失败重来之外的额外玩法变体

## 方案选择

### 方案 A：纯黑盒通关脚本

特点：

- 只看截图和固定输入时间表推进

问题：

- 对当前原型输入精度要求过高
- 一旦手感或帧节奏微调就容易失效
- 调试成本高，失败定位差

结论：

- 不采用

### 方案 B：灰盒主线驾驶器

特点：

- 真实输入推进
- 运行中读取房间名、HUD 步骤文本、玩家位置/速度作为导航依据
- 不直接作弊切房

优点：

- 能适配当前 Godot MCP 能力
- 定位失败点容易
- 比纯黑盒稳定，且仍保留“真实推进”意义

结论：

- 采用本方案

### 方案 C：混合方案（中途切房 + 终点黑盒）

特点：

- 前半段用脚本直接切房，后半段再真走

问题：

- 会削弱“完整主线通关验证”的价值
- 很容易退化成已有 GUT 的变体

结论：

- 不采用

## 总体设计

新增一个“灰盒主线驾驶器”测试入口，分为三层：

### 1. 输入层

职责：

- 统一发送短脉冲输入
- 所有输入都必须显式 `按下 -> 等待 -> 释放`
- 不允许长按不释放

形式：

- 基于 `godot_mcp` 的 `run_test_scenario` 或等价运行态输入能力
- 每段输入为最小动作片段，例如：
  - 右移 0.12s
  - 右移 + 跳跃
  - 贴地冲刺
  - 近身攻击

### 2. 观察层

职责：

- 读取当前运行态上下文
- 为策略层提供稳定快照

最小快照字段：

- 当前房间脚本或场景路径
- 当前 HUD `StepLabel`
- 当前 HUD `PromptLabel`
- 玩家 `global_position`
- 玩家 `velocity`
- 必要时读取少量门控状态或敌人存在性

### 3. 策略层

职责：

- 根据当前房间和 HUD 步骤决定下一段动作
- 每个房间只写最小推进策略，不做通用寻路

建议的阶段化策略：

- `TutorialRoom`
  - 推到上方平台
  - 穿过 dash 门槛
  - 靠近并攻击训练目标
  - 推到右侧出口
- `CombatTrialRoom`
  - 接近敌人
  - 通过攻击清敌
  - 推到出口
- `GoalTrialRoom`
  - 接近并清敌
  - 推到目标区
- `stage9` 主线房间链
  - 复用房间级最小推进策略
  - 通过房间名切换策略
- `stage10` 主线房间链
  - 主线优先，不主动进入支路
  - 按房间结构推进到挑战房再进入 demo 终点
- `stage11_demo_end_room`
  - 推到终点区
  - 读取 HUD 或主流程快照确认 `demo_completed`

## 测试设计

先补一条失败测试，再实现驾驶器。

### 新增测试目标

新增一条 Stage 11 自动化测试，保护：

- 从 `Main.tscn` 启动
- 使用灰盒主线驾驶器真实推进
- 最终进入 `stage11_demo_end_room`
- 触发 demo 完成反馈

### 测试失败输出

若失败，测试至少输出：

- 最后房间
- 最后 HUD 步骤文本
- 玩家最后坐标与速度
- 当前正在执行的策略步骤

这样可以把“失败在何处”直接记入测试输出，而不需要回头人工重放。

## 实现边界

本轮实现允许新增：

- 一个 Stage 11 灰盒主线自动化测试文件或扩展现有 Stage 11 测试
- 一个测试侧驾驶器辅助脚本
- 少量仅用于读取稳定快照的现有脚本接口补强

本轮实现不允许：

- 为了自动化方便而给生产代码新增“作弊切房”“强制完成”等入口
- 修改主线玩法节奏以迁就测试
- 把测试脚本变成第二套玩法控制系统

## 验证计划

本轮落地后至少执行：

- 新增的灰盒主线自动化测试，先红后绿
- 现有 `tests/stage11/test_stage_11_playable_demo_slice.gd`
- `godot --headless --path . --import`

## 风险

### 风险 1：输入抖动导致测试脆弱

应对：

- 统一用短脉冲
- 每段动作后都读取运行态快照确认是否到位
- 尽量用“达到某位置/某 HUD 步骤”而不是纯时间驱动

### 风险 2：策略脚本过大

应对：

- 每个房间只放最小推进策略
- 复用统一输入层与观察层
- 必要时拆成房间级辅助函数

### 风险 3：测试变成作弊

应对：

- 明确禁止直接 `transition_to_room()`
- 明确禁止瞬移和直接改状态
- 全程必须由真实输入推进

## 完成标准

满足以下条件才算本轮完成：

- 已存在一条可重复执行的灰盒主线自动化测试
- 测试从 `Main.tscn` 起步而不是中途切房
- 测试最终真实推进到 demo 终点并读到完成反馈
- 测试失败时能给出明确卡点上下文
- Stage 11 既有 GUT 不回归
