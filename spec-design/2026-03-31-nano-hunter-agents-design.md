# Nano Hunter — AGENTS.md 设计规范 (2026-03-31)

## 文档定位说明

本文档用于记录 `nano-hunter` 项目级 `AGENTS.md` 的设计依据、取舍过程与阶段性判断，回答“为什么这样制定这份规范”。

对应的 `AGENTS.md` 则是面向代理与后续 session 的执行规范，回答“从现在开始应如何照此执行”。

两者允许在内容上有阶段性重合，但职责不同：

- 本文档保留设计背景、范围判断与治理取舍
- `AGENTS.md` 保留当前生效的项目规则、流程与入口顺序

如果后续只是项目执行规则更新，应优先修改 `AGENTS.md`；如果治理思路、分层方式或规则来源发生变化，则应同时回写本文档。

## 1. 目标

为 `nano-hunter` 建立一份项目专属的 `AGENTS.md`，作为当前仓库内代理与后续开发 session 的项目级最高工作约定。该文件需要：

- 继承 `superpowers` 的核心方法论
- 结合 `nano-hunter` 当前处于 Godot 2D 类银河恶魔城原型期的实际情况
- 抛弃参考仓库 `angel-fallen` 中对本项目无用、过重或不兼容的内容
- 强制要求开发过程按时间线和状态文档留痕，保证后续重开 session 时可快速恢复上下文

---

## 2. 输入背景

当前 `nano-hunter` 仓库状态：

- Godot 4.6 工程已初始化
- 当前启用插件为 `better-terrain`、`godot_mcp`、`gut`
- 仓库中已有 `spec-design/` 设计文档
- 目前尚无成熟的游戏代码目录、测试目录和 CI 流程
- 项目目标是制作一款 2D 类银河恶魔城游戏，当前阶段优先做可玩的垂直切片

因此，`AGENTS.md` 不应照搬成熟项目模板，不应假定：

- 已存在完整的 `scenes/`、`scripts/`、`tests/`、CI 结构
- 已存在复杂的单例体系、数据校验脚本、发布门禁脚本
- 当前就要实现完整中后期系统

---

## 3. AGENTS.md 的定位

`AGENTS.md` 应被定义为项目级最高工作约定，主要负责：

1. 说明 `nano-hunter` 当前是什么项目、处于什么阶段
2. 规定开发流程应如何遵循 `superpowers`
3. 规定文档留痕是强制要求，而非可选习惯
4. 作为新 session 或新代理进入项目时的第一阅读入口

优先级说明：

- 用户直接要求优先
- 本项目 `AGENTS.md` 高于通用默认习惯
- 技能工作流在本项目约束下执行

## 4. 工作流策略：混合版 Superpowers

本项目采用“混合版 superpowers 工作流”。

### 4.1 大功能

满足任一条件，按大功能处理：

- 新增或重做核心玩法系统
- 修改角色控制、战斗循环、状态机、敌人 AI、关卡推进、能力门控
- 新增或重做主流程 UI
- 新增项目级目录规范、资源规范、测试规范、存档规范
- 任何明显影响后续开发方向的工作

大功能必须走完整流程：

1. `brainstorming` 或等效设计确认
2. 设计文档确认或补写
3. implementation plan
4. 实现
5. 验证
6. 更新进度文档

### 4.2 小改动

以下工作可按小改动处理：

- 小范围配置调整
- 插件启停和工程整理
- 小型目录修正
- 单脚本的小 bug 修复
- 不影响整体架构的辅助性修改

小改动允许轻量流程，但仍必须：

1. 先写明目标和影响范围
2. 再实施修改
3. 做最小必要验证
4. 更新进度文档

### 4.3 升级条件

即使是表面上的小改动，只要出现以下情况，也必须升级为大功能流程：

- 需求边界不清
- 会影响多个子系统
- 需要新建多个核心文件或目录
- 可能改变后续玩法方向
- 设计文档与现实实现明显冲突

### 4.4 原型期原则

`nano-hunter` 当前阶段默认遵循：

- 优先垂直切片，而不是功能铺量
- 优先可玩，而不是完整
- 默认不直接展开完整元素池、完整姿态池、完整区域结构
- 当前实现目标聚焦教程区和基础战斗探索循环

## 5. 文档留痕策略

### 5.1 总原则

所有开发活动都必须留痕。代码不是唯一真源，设计与进度文档同样属于项目真源的一部分。

每次记录至少要回答：

- 做了什么
- 为什么做
- 影响了什么
- 结果如何
- 下一步是什么
- 有无阻塞或风险

### 5.2 文档结构

项目文档采用双层结构：

- `spec-design/`
  - 存放设计文档、玩法设计、系统设计
  - 现有文档继续沿用
  - 如实现导致设计变化，必须同步回写

- `docs/progress/status.md`
  - 当前项目状态真源
  - 记录当前阶段、当前里程碑、进行中事项、最近完成项、阻塞项、下一步

- `docs/progress/timeline.md`
  - 全项目关键开发时间线
  - 用于快速回顾项目演进过程

- `docs/progress/YYYY-MM-DD.md`
  - 当天开发日志
  - 记录当日具体改动、验证结果、遗留问题、下一步

### 5.3 更新频率

- 大功能：
  - 开始前确认设计入口
  - 关键设计确认后更新 spec
  - 实现过程中至少更新一次当天日志
  - 完成后必须更新 `status.md` 和 `timeline.md`

- 小改动：
  - 完成后至少更新一次当天日志
  - 若影响方向、优先级或后续安排，额外更新 `status.md`

### 5.4 文档优先级

若信息冲突，优先判断顺序为：

1. 当前代码和可运行结果
2. 最新的进度文档
3. 最新的设计文档
4. 更早的聊天上下文

### 5.5 强制要求

- 未更新进度文档的开发不算完成
- 未记录验证结果的改动不算真正交付
- 发现设计与实现偏离时，必须先记文档再继续推进

## 6. 当前阶段工程约定

### 6.1 项目阶段与目标

当前项目阶段为 `vertical slice / 原型期`。

此部分属于可演进的阶段性规范，应随里程碑推进而更新，但不因零碎小任务频繁改动。

当前默认目标：

- 主场景
- 玩家基础移动
- 跳跃 / 冲刺 / 基础攻击
- 2 个姿态以内
- 2-3 个元素以内
- 最小 HUD
- 最小可回溯关卡验证

其余中后期内容保留在设计文档中，不默认进入实现。

### 6.2 目录约定

`AGENTS.md` 中应定义目标目录，而不是假定当前目录已成熟：

- `scenes/`：游戏场景
- `scripts/`：核心脚本
- `assets/`：美术与音频资源
- `tests/`：GUT 测试
- `spec-design/`：设计文档
- `docs/progress/`：进度与时间线文档

原则：

- 目录扩展小步前进
- 没有明确需求时，不提前铺设复杂结构
- 命名优先服务当前原型开发

### 6.3 验证约定

当前阶段只保留真实有效的验证要求：

- 工程配置改动后，至少做一次 Godot 无头启动检查
- 脚本或场景改动后，优先做最小可运行验证
- 若已有 GUT 测试，则优先运行最接近的测试
- 修 bug 时，优先补最小回归测试
- 未记录验证结果，不应声称“完成”

### 6.4 插件约定

当前默认启用插件：

- `better-terrain`
- `godot_mcp`
- `gut`

其他插件视为“已安装但当前阶段不启用”。

若后续新增或重新启用插件，必须记录：

- 启用日期
- 启用目的
- 影响范围

### 6.5 提交约定

提交约定应适配原型期：

- 每次提交围绕一个明确进展点
- 提交信息使用“中文 + English”
- 提交前至少确认：
  - 改动已验证
  - 进度文档已更新
  - 未提交无关噪音文件

若只是中途探索但结果未稳定，可不立即提交，但仍需写入当天日志。

### 6.6 Session 接续约定

新 session 进入项目时，建议阅读顺序：

1. `AGENTS.md`
2. `docs/progress/status.md`
3. 最近一篇 `docs/progress/YYYY-MM-DD.md`
4. 相关 `spec-design/` 文档

不应只凭聊天末尾几句直接进入编码。

## 7. 对参考 AGENTS.md 的取舍结论

参考仓库 `angel-fallen` 中以下内容对 `nano-hunter` 当前阶段不适合直接继承：

- 已存在主场景、完整目录、成熟单例体系的假设
- 依赖现成 Python 校验脚本、资源 acceptance、CI 发布门禁的重流程
- 面向成熟项目的测试和资源基线管理要求
- 与当前仓库无关的资产自动化管线和外部 API 工作流
- 已成型项目的详细数据目录和资源 ID 约束

应保留并改造的思想包括：

- 项目专属规则优先
- 验证先于完成宣称
- 注释与命名需统一
- 提交需有明确边界
- 文档要能够帮助后续代理快速接手

## 8. 落地结果
