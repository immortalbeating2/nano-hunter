# Nano Hunter — AGENTS.md 设计规范 (2026-03-31)

## 文档定位说明

本文档用于记录 `nano-hunter` 项目级 `AGENTS.md` 的设计依据、取舍过程与阶段性判断，回答“为什么这样制定这份规范”。

对应的 `AGENTS.md` 则是面向代理与后续 session 的执行规范，回答“从现在开始应如何照此执行”。

两者允许在内容上有阶段性重合，但职责不同：

- 本文档保留设计背景、范围判断与治理取舍
- `AGENTS.md` 保留当前生效的项目规则、流程与入口顺序

如果后续只是项目执行规则更新，应优先修改 `AGENTS.md`；如果治理思路、分层方式或规则来源发生变化，则应同时回写本文档。

## 1. 目标

为 `nano-hunter` 建立一份项目专属的 `AGENTS.md`，作为当前仓库内代理与后续开发 session 的项目级最高工作约定。该文件需要：

- 继承 `superpowers` 的核心方法论
- 结合 `nano-hunter` 当前处于 Godot 2D 类银河恶魔城原型期的实际情况
- 抛弃参考仓库 `angel-fallen` 中对本项目无用、过重或不兼容的内容
- 强制要求开发过程按时间线和状态文档留痕，保证后续重开 session 时可快速恢复上下文

## 2. 输入背景

当前 `nano-hunter` 仓库状态：

- Godot 4.6 工程已初始化
- 当前默认可用插件为 `godot_mcp`、`gut`，`better-terrain` 保留在仓库中但当前不启用
- 仓库中已有 `spec-design/` 设计文档
- 目前尚无成熟的游戏代码目录、测试目录和 CI 流程
- 项目目标是制作一款 2D 类银河恶魔城游戏，当前阶段优先做可玩的垂直切片

因此，`AGENTS.md` 不应照搬成熟项目模板，不应假定：

- 已存在完整的 `scenes/`、`scripts/`、`tests/`、CI 结构
- 已存在复杂的单例体系、数据校验脚本、发布门禁脚本
- 当前就要实现完整中后期系统

## 3. AGENTS.md 的定位

`AGENTS.md` 应被定义为项目级最高工作约定，主要负责：

1. 说明 `nano-hunter` 当前是什么项目、处于什么阶段
2. 规定开发流程应如何遵循 `superpowers`
3. 规定文档留痕是强制要求，而非可选习惯
4. 作为新 session 或新代理进入项目时的第一阅读入口
5. 规定面向用户与项目文档的默认交互语言，减少后续 session 的语言风格漂移

优先级说明：

- 用户直接要求优先
- 本项目 `AGENTS.md` 高于通用默认习惯
- 技能工作流在本项目约束下执行

交互语言约定：

- 面向用户的说明、过程更新、提权理由、子代理任务描述和项目文档默认使用中文
- 提交信息继续使用“中文 + English”
- 若工具、系统固定 UI、第三方命令输出或官方术语无法稳定中文化，可保留原文，并在必要时补充中文说明

## 4. 工作流策略：混合版 Superpowers

本项目采用“混合版 superpowers 工作流”。

### 4.1 大功能

满足任一条件，按大功能处理：

- 新增或重做核心玩法系统
- 修改角色控制、战斗循环、状态机、敌人 AI、关卡推进、能力门控
- 新增或重做主流程 UI
- 新增项目级目录规范、资源规范、测试规范、存档规范
- 任何明显影响后续开发方向的工作

大功能必须走完整流程：

1. `brainstorming` 或等效设计确认
2. 设计文档确认或补写
3. implementation plan
4. 实现
5. 验证
6. 更新进度文档

### 4.2 小改动

以下工作可按小改动处理：

- 小范围配置调整
- 插件启停和工程整理
- 小型目录修正
- 单脚本的小 bug 修复
- 不影响整体架构的辅助性修改

小改动允许轻量流程，但仍必须：

1. 先写明目标和影响范围
2. 再实施修改
3. 做最小必要验证
4. 更新进度文档

### 4.3 升级条件

即使是表面上的小改动，只要出现以下情况，也必须升级为大功能流程：

- 需求边界不清
- 会影响多个子系统
- 需要新建多个核心文件或目录
- 可能改变后续玩法方向
- 设计文档与现实实现明显冲突

### 4.4 原型期原则

`nano-hunter` 当前阶段默认遵循：

- 优先垂直切片，而不是功能铺量
- 优先可玩，而不是完整
- 默认不直接展开完整元素池、完整姿态池、完整区域结构
- 当前实现目标聚焦教程区和基础战斗探索循环

## 5. 文档留痕策略

### 5.1 总原则

所有开发活动都必须留痕。代码不是唯一真源，设计与进度文档同样属于项目真源的一部分。

每次记录至少要回答：

- 做了什么
- 为什么做
- 影响了什么
- 结果如何
- 下一步是什么
- 有无阻塞或风险

### 5.2 文档结构

项目文档采用双层结构：

- `spec-design/`
  - 存放设计文档、玩法设计、系统设计
  - 现有文档继续沿用
  - 如实现导致设计变化，必须同步回写

- `docs/progress/status.md`
  - 当前项目状态摘要入口
  - 需要和 `docs/progress/timeline.md` 及当日日志一起阅读
  - 记录当前阶段、当前里程碑、进行中事项、最近完成项、阻塞项、下一步

- `docs/progress/timeline.md`
  - 全项目关键开发时间线
  - 用于快速回顾项目演进过程

- `docs/progress/YYYY-MM-DD.md`
  - 当天开发日志
  - 记录当日具体改动、验证结果、遗留问题、下一步

### 5.3 更新频率

- 大功能：
  - 开始前确认设计入口
  - 关键设计确认后更新 spec
  - 实现过程中至少更新一次当天日志
  - 完成后必须更新 `status.md` 和 `timeline.md`

- 小改动：
  - 完成后至少更新一次当天日志
  - 若影响方向、优先级或后续安排，额外更新 `status.md`

### 5.4 文档优先级

若信息冲突，优先判断顺序为：

1. 当前代码和可运行结果
2. 最新的进度文档
3. 最新的设计文档
4. 更早的聊天上下文

### 5.5 强制要求

- 未更新进度文档的开发不算完成
- 未记录验证结果的改动不算真正交付
- 发现设计与实现偏离时，必须先记文档再继续推进

### 5.6 Subagent / Multi-Agent 与 Delegation Log

本项目允许使用 `subagent` 与 `multi-agent` 协作，但应保持轻量，并服务于当前原型期的效率与可追溯性。

约定如下：

- 默认由主代理单线推进；只有在任务边界清楚、可隔离或需要独立审查时，才启用 `subagent`
- 当存在 2 个以上互不阻塞、写入范围可分离、且已经有明确计划的任务时，才启用 `multi-agent` 并行
- 适合 `subagent` 的工作包括：独立调查、spec review、quality review、文档整理、小范围实现
- 适合 `multi-agent` 的工作包括：按文件或目录明确分工的并行实现，例如场景、玩家脚本、HUD、验证文档分别推进
- 以下情况不应委派或不应并行：需求未收敛、核心设计仍在变化、多个任务会改同一批文件、当前下一步强依赖单个结果、单点调试尚未定位
- 主代理始终负责设计确认、任务拆分、结果整合、最终验证和对用户交付；子代理只负责被分配的局部任务
- 2026-04-26 起，项目已将可复用 agent 角色池写入 `.codex/agent/`，并在 `AGENTS.md` 中固定当前生效规则
- 当前角色池采用 6 类通用游戏项目角色：`design`、`architecture`、`gameplay`、`content`、`qa`、`production`
- 并行上限默认 `max_threads = 4`、层级上限默认 `max_depth = 1`；角色池固定不等于每个任务都要全量启用
- 本文档保留早期“暂不固定 roster”的设计背景；当前执行口径以 `AGENTS.md` 与 `.codex/agent/` 为准

为了确保追溯性，不单独新建复杂的委派文档体系，而是在 `docs/progress/YYYY-MM-DD.md` 中按需增加 `Delegation Log` 小节。

每条记录至少包括：

- 用途
- 范围
- 模式：`subagent` 或 `multi-agent`
- 结果
- 收口：是否经过主代理整合、review 或 verification

## 6. 当前阶段工程约定

### 6.1 项目阶段与目标

当前项目阶段为 `vertical slice / 原型期`。

此部分属于可演进的阶段性规范，应随里程碑推进而更新，但不因零碎小任务频繁改动。

当前默认目标：

- 主场景
- 玩家基础移动
- 跳跃 / 冲刺 / 基础攻击
- 2 个姿态以内
- 2-3 个元素以内
- 最小 HUD
- 最小可回溯关卡验证

其余中后期内容保留在设计文档中，不默认进入实现。

### 6.2 阶段推进与可试玩检查点

第一版可玩原型不按自然时间长短切阶段，而按“是否已经可看、可玩、可调”切分为 5 个可试玩检查点。

总原则：

- 阶段长度按退出条件定义，不按几天或几周定义
- 每个阶段都必须同时满足：`可看`、`可玩`、`可调`
- 未达到可试玩标准，不进入下一阶段
- 若某阶段手感或方向明显不对，允许回退到上一阶段继续调整

#### 阶段 1：可启动骨架

- 目标：让项目可以直接启动进入主场景，建立后续一切玩法验证的稳定入口
- 至少包含：`Main.tscn`、`run/main_scene`、测试房间、玩家出生点、相机与基础碰撞
- 当前可见结果：游戏窗口能打开，玩家或占位体能出现在场景里
- 当前可调项：分辨率、相机基础策略、房间尺寸、场景比例
- 退出条件：项目可直接启动进入场景，足以承接后续移动手感开发

#### 阶段 2：基础移动手感

- 目标：验证“走、停、跳、落地”的基础手感
- 至少包含：移动、跳跃、落地、基础状态切换
- 当前可见结果：玩家能在简单平台环境中跑跳
- 当前可调项：速度、加速度、减速度、跳跃高度、滞空感
- 退出条件：移动与跳跃手感已可稳定体验，并能支持后续战斗接入

#### 阶段 3：基础战斗手感

- 目标：验证最小攻击循环是否成立
- 至少包含：普攻、攻击判定、受击目标、基础受击反馈
- 当前可见结果：玩家能打中假人或基础敌人
- 当前可调项：攻击前摇、后摇、攻击范围、命中反馈、受击停顿感
- 退出条件：最小攻击循环已可体验，并有足够反馈支持方向判断

#### 阶段 4：最小能力差异

- 目标：验证至少 1 组能力差异是否真的带来玩法价值
- 至少包含：1 组姿态切换或元素差异
- 当前可见结果：角色不再是单一动作模板，开始出现功能或节奏差异
- 当前可调项：切换节奏、能力用途、战斗与探索中的价值占比
- 退出条件：能力差异已具备实际体验意义，而不是纯数值或纯视觉变化

#### 阶段 5：教程区垂直切片

- 目标：把前述内容串成一段短但完整的可玩流程
- 至少包含：小区域、基础敌人或障碍、简单门控、HUD 占位、起点到终点的短流程
- 当前可见结果：接近真正游戏雏形的第一版教程区
- 当前可调项：引导节奏、空间长度、反馈密度、难度曲线
- 退出条件：玩家可从起点走到终点，并对整体节奏和方向给出明确反馈

### 6.3 阶段文档字段与 status 表达

每次阶段推进至少要在文档中明确以下字段：

- `阶段名称`
- `阶段目标`
- `当前可见结果`
- `当前可玩结果`
- `当前可调项`
- `退出条件`
- `当前资产状态`

`docs/progress/status.md` 作为摘要入口，至少应固定维护：

- `Current Stage`
- `Stage Goal`
- `Playable Now`
- `Adjustable Now`
- `Exit Criteria`
- `Asset Status`
- `Next Stage`

要求：

- 如果当前还没有可试玩版本，应在 `Playable Now` 中明确写出“当前不可试玩”，而不是省略
- 每完成一个阶段，必须同步更新 `status.md` 和当日日志
- 当日日志应额外记录：本阶段验证结果、调参重点、用户反馈结论

### 6.4 原型期资产策略

原型期的核心原则是：不让正式资产生产阻塞玩法验证。

资产分层如下：

- `占位资产`
  - 纯色块、简单几何形、临时图标
  - 用于最快验证玩法与空间关系

- `免费替代资产`
  - 免费 tileset、免费角色、免费图标或特效
  - 用于提升可读性与可玩性，但不视为最终风格承诺

- `AI 临时资产`
  - 用于关键视觉探索和缺口补位
  - 适合角色轮廓、能力识别、HUD 方向、教程区氛围等方向收敛

- `正式项目资产`
  - 在玩法与视觉方向稳定后，再逐步替换前面三类资源

按阶段安排：

- `阶段 1`
  - 以占位资产为主
  - 允许少量免费 tileset
  - AI 只做氛围或比例参考，不做批量正式产出

- `阶段 2`
  - 玩家与平台继续以占位或免费资源为主
  - AI 可探索角色比例、轮廓和动作方向

- `阶段 3`
  - 攻击目标、hit VFX 可先用占位或免费资源
  - AI 可探索攻击反馈和特效方向

- `阶段 4`
  - 能力差异必须先做到视觉可读，即使没有正式美术
  - 可先用颜色、轮廓、图标或特效来区分

- `阶段 5`
  - 教程区允许混合使用占位、免费与 AI 临时资产
  - 目标是看得懂、玩得下去、能提出修改意见，而不是最终美术完成

切换条件：

- 只有当某个玩法模块已验证成立，且短期内不会频繁推翻时，才开始将临时资源替换成更正式资产
- 如果玩法仍在剧烈变化，应优先继续使用占位或免费替代，不提前进入正式资产生产

### 6.5 目录约定

`AGENTS.md` 中应定义目标目录，而不是假定当前目录已成熟：

- `scenes/`：游戏场景
- `scripts/`：核心脚本
- `assets/`：美术与音频资源
- `tests/`：GUT 测试
- `spec-design/`：设计文档
- `docs/progress/`：进度与时间线文档

原则：

- 目录扩展小步前进
- 没有明确需求时，不提前铺设复杂结构
- 命名优先服务当前原型开发

### 6.6 验证约定

当前阶段只保留真实有效的验证要求：

- 工程配置改动后，至少做一次 Godot 无头启动检查
- 脚本或场景改动后，优先做最小可运行验证
- 若已有 GUT 测试，则优先运行最接近的测试
- 修 bug 时，优先补最小回归测试
- 未记录验证结果，不应声称“完成”

### 6.7 插件约定

当前默认启用插件：

- `godot_mcp`
- `gut`

`better-terrain` 保留在仓库中，但当前阶段不启用。其他插件视为“已安装但当前阶段不启用”。

若后续新增或重新启用插件，必须记录：

- 启用日期
- 启用目的
- 影响范围

### 6.8 提交约定

提交约定应适配原型期：

- 每次提交围绕一个明确进展点
- 提交信息使用“中文 + English”
- 提交前至少确认：
  - 改动已验证
  - 进度文档已更新
  - 未提交无关噪音文件

若只是中途探索但结果未稳定，可不立即提交，但仍需写入当天日志。

### 6.9 Session 接续约定

新 session 进入项目时，建议阅读顺序：

1. `AGENTS.md`
2. `docs/progress/status.md`
3. 最近一篇 `docs/progress/YYYY-MM-DD.md`
4. 相关 `spec-design/` 文档

不应只凭聊天末尾几句直接进入编码。

## 7. 对参考 AGENTS.md 的取舍结论

参考仓库 `angel-fallen` 中以下内容对 `nano-hunter` 当前阶段不适合直接继承：

- 已存在主场景、完整目录、成熟单例体系的假设
- 依赖现成 Python 校验脚本、资源 acceptance、CI 发布门禁的重流程
- 面向成熟项目的测试和资源基线管理要求
- 与当前仓库无关的资产自动化管线和外部 API 工作流
- 已成型项目的详细数据目录和资源 ID 约束

应保留并改造的思想包括：

- 项目专属规则优先
- 验证先于完成宣称
- 注释与命名需统一
- 提交需有明确边界
- 文档要能够帮助后续代理快速接手
