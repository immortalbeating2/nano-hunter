# 阶段 7：短链路主流程串联设计

## 文档定位

本文档用于记录 `nano-hunter` 在 `阶段 7：短链路主流程串联` 的第一轮边界、关键决策与验证目标，确保本轮把当前已成立的 `TutorialRoom -> CombatTrialRoom` 推进成一条真正可走通的最小主流程，而不是提前扩成通用房间图、完整地图系统或正式 checkpoint 框架。

## 背景

截至 `2026-04-23`，项目已经完成：

- 阶段 1：可启动骨架
- 阶段 2：基础移动手感
- 阶段 3：基础战斗手感
- 阶段 4：最小能力差异
- 阶段 5：教程区垂直切片
- 阶段 6：最小真实战斗循环

当前 `main` 已具备以下稳定基线：

- `Main` 默认进入 `TutorialRoom`
- 玩家已具备移动、跳跃、地面 `dash`、地面普通攻击、生命、受击、无敌与 `defeated`
- `TutorialRoom` 已完成低压教学切片
- `CombatTrialRoom` 已完成第一次真实战斗压力验证
- 最小 HUD 已能真实读取提示、生命值与 `dash` 状态
- 阶段 1-6 自动化测试均已建立

当前最大的玩法缺口已经不再是“能不能教学”或“有没有真实压力”，而是“这些内容能否被串成一条短主流程”。因此阶段 7 的目标不是新增新能力，而是把现有能力与现有压力源放进一条连续推进的主线链路中。

## 本轮目标

本轮固定只做以下内容：

- 把当前流程固定为 `TutorialRoom -> CombatTrialRoom -> GoalTrialRoom`
- 新增 1 个 `GoalTrialRoom`，作为混合门控目标房
- 让 `Main` 从阶段 6 的定向切房逻辑，提升为支持三段顺序流转的最小主流程控制
- 让 HUD 能随三段流程切换上下文提示
- 保持结果满足：
  - 可看：玩家能看懂这已经是一条连续主流程，而不是单房间演示
  - 可玩：玩家能顺序完成教学、实战、目标房的短链路
  - 可调：门控布局、提示节奏、房间切换与目标完成条件都可继续微调

## 明确不在本轮范围内

本轮明确不做：

- 大地图
- 存档
- 正式 checkpoint 系统
- 多分支路线
- 第二类敌人
- 新能力
- 资源收集循环
- 背包或钥匙系统
- 完整房间图系统

## 关键实现决策

### 1. 主流程固定为三段顺序链路

- 本轮不压缩为“两段连房”
- 也不直接跳到通用房间系统
- 固定为：
  - `TutorialRoom`
  - `CombatTrialRoom`
  - `GoalTrialRoom`

这样做的原因是：

- 能在最小范围内验证“连续推进”是否成立
- 复用阶段 5 与阶段 6 已验证的两个房间
- 为阶段 8 的系统稳固保留清晰前置条件

### 2. 第三段固定为混合门控目标房

- `GoalTrialRoom` 不是纯平台房
- 也不是纯战斗房
- 玩家必须在房间中同时处理：
  - 一次空间门控
  - 至少一次最小攻击型交互或压力解除
  - 一次明确的目标到达

这样做的原因是：

- 阶段 7 需要验证“能力组合”而不是单点能力
- 能避免把第三段做成“战斗房复刻”或“教程房复刻”

### 3. 房间切换能力仍保持最小，不扩成正式房间系统

- `Main` 继续只负责：
  - 当前房间实例化与切换
  - 玩家生成与重置
  - HUD 绑定
- 房间内部门控与目标判定仍由房间脚本自己管理

这样做的原因是：

- 本轮只是验证短链路主流程是否成立
- 还不到引入通用房间图、存档点与路径管理的时候

### 4. HUD 升级为主流程分段提示，但保持最小

- HUD 继续保留提示区与战斗面板
- 提示区需要明确支持：
  - 教学房提示
  - 实战房提示
  - 目标房提示
  - 短链路完成提示
- 仍不引入独立结算 UI 或复杂前台系统

这样做的原因是：

- 阶段 7 的目标是让玩家“知道自己正在推进一条主流程”
- 不需要为了这件事提前扩出完整 UI 架构

## 代理协作硬约束

阶段 6 已证明“只写建议、不写硬约束”时，实际实现 session 很容易回到主代理单线推进。阶段 7 起，本项目在满足条件时把 `subagent / multi-agent` 视为实现约束，而不是软建议。

### 必须启用代理协作的条件

只要同时满足以下条件，就必须启用代理协作：

- 存在 2 个以上子任务
- 子任务写入范围可隔离
- 子任务验证可独立
- 下一步不强依赖单一阻塞结果

### 本轮默认拆分

若实现前复核后条件仍成立，阶段 7 默认采用 `multi-agent`，推荐拆分为：

- 代理 A：`Main` 与三段房间流转契约
- 代理 B：`GoalTrialRoom` 与门控流程
- 代理 C：HUD 分段提示、Stage 7 GUT、文档留痕

### 允许降级但不允许默认回退

若实现时发现多个子任务会持续同时改同一批核心文件，则可以降级为：

- 至少 1 个 `subagent` 处理非阻塞侧任务
- 主代理只保留阻塞主线整合

只有在以下例外条件成立时，才允许本轮不启用 `multi-agent`：

- 需求仍未收敛
- 单点调试尚未定位
- 写入范围无法隔离

若发生例外，必须在 `docs/progress/YYYY-MM-DD.md` 的 `Delegation Log` 中写明原因、范围与后续重新评估时机。

## 验证标准

本轮至少需要验证：

1. `Main` 能按顺序从 `TutorialRoom -> CombatTrialRoom -> GoalTrialRoom` 推进
2. `GoalTrialRoom` 满足主房间契约，并提供独立出生点与完成条件
3. 目标房初始门控关闭，满足条件后解锁
4. HUD 能随三段流程切换提示
5. 玩家在 `CombatTrialRoom` 中失败时，仍只重置当前战斗房，不回滚整条链路
6. 阶段 1 / 2 / 3 / 4 / 5 / 6 自动化测试继续保持绿色

## 风险与注意项

- 本轮最容易失控的方向是：把房间切换顺势扩成正式房间图、checkpoint 或地图系统
- `GoalTrialRoom` 若只做纯平台或纯战斗，会削弱阶段 7 的验证价值
- 若继续把代理协作只写成“建议”，很可能再次在实际实现时被跳过；因此本轮必须在计划和日志中都写成硬约束
- 清理本轮 worktree 时，仍需遵守 `AGENTS.md` 中已写明的顺序：先定位并关闭指向该 worktree 的 Godot / VS Code / 终端等进程，再删物理目录
